### 1. 主檔層：先定義「有哪些異常」

| 資料表 | 說明 |
|---|---|
| cate_main | 主分類，例如製程事故、設備故障、品質異常 |
| cate_mid | 中分類，例如 COBBLE異常、軋輥異常、傳動異常 |
| cate_detail | 細分類，例如 ROD、BIC、BAR |
| equipment | 設備或站點，例如 HRM、KB1、KB2、FB、TMB |
| state | 狀況項目，例如打滑、軸承破裂、訊號 loss |
| state_mapping | 把分類、設備、狀況關聯起來，讓查詢可以收斂 |

### 2. SOP 層：定義「發生異常時怎麼處理」

| 資料表 | 說明 |
|---|---|
| sop_main | 一種設備 + 一種狀況，對應一份 SOP |
| sop_step | SOP 的流程節點，用 parent_step_id 表示樹狀流程 |
| attachment | SOP 圖片、正常/異常照片、異常報告附件 |

### 3. 監控規則層：定義「哪些步驟可以自動判斷」

| 資料表 | 說明 |
|---|---|
| monitor_function | 可綁定的監控功能，例如加熱爐在爐時間、HRM水量 |
| parameter_spec | 每個監控項目的標準值、上下限、允差與單位 |
| sensor_snapshot | 異常事件發生時的實際參數值與判斷結果 |

### 4. 異常歷史層：定義「每次真的發生了什麼」

| 資料表 | 說明 |
|---|---|
| abnormal_event | 異常事件主檔，記錄時間、設備、停機、原因、處置 |
| event_step_check | 該事件依 SOP 每個步驟的確認結果 |

## 三、資料可以怎麼應用

### 應用一：SOP 查詢系統

使用者選擇：主分類 → 中分類 → 細分類 → 設備 → 狀況項目。
系統用 `state_mapping` 找到 `state_id`，再用 `sop_main` 找到 SOP，最後用 `sop_step` 畫出流程圖。

### 應用二：SOP 節點自動判斷

`SOP_STEP.monitor_id` 若有綁定監控項目，就可以去 `sensor_snapshot` 取實際值，再和 `parameter_spec` 比對。

範例：

- SOP 節點：確認加熱爐時間與溫度。
- 監控項目：加熱爐在爐時間。
- 標準：依鋼種有效加熱時間 ±30 分鐘。
- 實際值：事件快照中的 effective_heating_time_min。
- 結果：正常 → 節點顯示綠色；異常 → 節點顯示紅色並提示原因。

### 應用三：異常管理看板

用 `abnormal_event` 可以做：

- 每月異常件數趨勢。
- Top 10 異常設備。
- Top 10 異常狀況。
- 平均停機時間 MTTR。
- 嚴重度 A/B/C 分布。
- 原因分類 Pareto。

### 應用四：歷史案例推薦

當新事件發生時，用以下條件找相似案例：

1. 同設備。
2. 同狀況項目。
3. 同鋼種或相近產品規格。
4. 相同原因分類。
5. 曾成功復機且停機時間較短。

系統可推薦過去有效的 `action_summary`。

### 應用五：SOP 改善

用 `event_step_check` 找出：

- 哪些 SOP 步驟最常判定異常。
- 哪些步驟最常需要人工補資料。
- 哪些設備的處理時間特別長。
- 哪些異常重複發生但 SOP 沒有改善。

這可以變成期末專題的亮點：**不只是查 SOP，而是讓 SOP 隨著異常處理經驗持續變好。**
